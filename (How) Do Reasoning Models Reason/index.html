Section 1

研究动机与问题背景

1.1 核心问题：大语言模型的两类思维

作者将认知心理学中的 Dual Process Theory（双过程理论，Kahneman, 2011）映射到 LLM 上：

🧠 System 1 — LLM 已擅长

  • 快速、直觉、自动化的 token 生成
  • 风格模仿、语言流畅性
  • 知识检索与模式匹配
  • 基于预训练语料的"近似检索"

🔬 System 2 — 仍是 "Achilles Heel"

  • 事实性（Factuality）
  • 逻辑推理（Reasoning）
  • 规划（Planning）
  • 可论证的鲁棒泛化

本文的核心问题是：最近兴起的大推理模型（Large Reasoning Models, LRMs）——如 OpenAI o1、DeepSeek R1——是否真的具备了 System 2 能力？ 它们的表现提升来自真正的推理，还是仅仅将验证信号"编译"进了更强大的近似检索？

1.2 为什么需要这篇 Perspective？

📌 研究动机：对抗叙事泡沫

作者指出，围绕 LRM 的主流叙事存在严重的拟人化偏差：人们倾向于将 LLM 生成的中间 token 序列称为 "Chain of Thought"（思维链）或 "reasoning trace"（推理轨迹），仿佛这些 token 确实反映了模型的"思考过程"。本文的目标是提供一个去拟人化（de-anthropomorphized）的统一视角，批判性地审视 LRM 的能力来源、错觉与局限。

🔑 核心主张： LRM 的进步可以被理解为——在 test time、train time 或 distillation time——利用外部验证器（verifier）信号，将"推理"迭代编译为"检索"（iteratively compiling reasoning into retrieval）。这意味着 LRM 并非真正学会了推理，而是将验证器的判别能力内化为了更强的生成先验。

1.3 论文的研究范畴

这是一篇 PerspectiveSurvey 性质的文章（非实验论文），旨在：

  • 系统梳理构建 LRM 的两类正交技术：Test-time inference scaling 与 Post-training on derivational traces
  • 建立理解 LRM 的概念框架：能力评估、成本效益、与基础 LLM 的本质差异、"推理 vs. 检索"的辨析
  • 对"中间 token 即推理轨迹"这一拟人化假设进行彻底批判
  • 提出 LRM 在 LLM-Modulo 混合框架中的最适角色
Section 2

数学表示与概念建模

2.1 Token 预测的基础形式

LLM 的基本运作可以描述为一个自回归 token 生成过程。给定前缀 $x_{1:t-1}$，模型输出下一个 token 的概率分布：

$$P(x_t \mid x_{1:t-1}) = \text{softmax}\big(f_\theta(x_{1:t-1})\big)$$

其中 $f_\theta$ 是参数为 $\theta$ 的 transformer 网络。训练目标是最大化训练语料中 token 序列的似然。这是一个纯粹的 System 1 过程：每一步都只是选择当前最可能的 token。

2.2 Prompt Augmentation 的 Skolem 函数形式化

这是本文最核心的数学贡献——将 LRM 的"推理"统一形式化为一个 Prompt Augmentation 问题：

🎯 核心形式化

给定一个任务提示 $T$，存在某个 Prompt Augmentation $\text{PA}$，使得：

$$\Pr\big(\text{Sol}(\text{LLM}(T + \text{PA}),\, T)\big) \;>\; \Pr\big(\text{Sol}(\text{LLM}(T),\, T)\big)$$

其中 $\text{LLM}(x)$ 表示 LLM 以 $x$ 为输入时的输出 completion，$\text{Sol}(y, T)$ 是一个二元谓词（可借助验证器实现），判断 $y$ 是否包含任务 $T$ 的正确解。

这个不等式揭示了一个关键洞见：几乎所有提升 LLM 推理能力的技巧，本质上都是在为每个 $T$ 寻找合适的 $\text{PA}$。

📐 "Holy Grail" 的定义：学习 Skolem 函数 $f_{\text{skolem}}$，使得对于任意 $T$，有 $\text{PA} = f_{\text{skolem}}(T)$ 能够最大化解的正确概率。Zero-shot CoT、Few-shot CoT、Post-training with RL 等方法，都是对这个 Skolem 函数的不同近似方式。

2.3.1 第一类：Test-Time Inference Scaling

在推理时不直接输出最可能的答案，而是让模型做更多工作。核心思想：并非所有问题都需要同等努力（2 位数加法 vs. 4 位数加法）。历史先例包括 limited-depth min-max 搜索、实时 A*、动态规划、MCTS。

1

Self-Consistency

生成 $k$ 个候选解，选择出现频率最高的答案。Wang et al., 2023。无正确性理论保证。

2

Verification

LLM-as-verifier（已知不可靠）、Learned Verifier（需训练）、External Sound Verifier（有保证但领域受限）。

3

LLM-Modulo

验证器对错误解提供反馈 $\rightarrow$ LLM 重新生成更优候选。Kambhampati et al., 2024。

4

Search

MCTS 等搜索算法引导探索 $\rightarrow$ 类似 AlphaGo 的 generate-test 循环。

2.3.2 第二类：Post-Training on Derivational Traces

作者刻意使用 "Derivational Traces"（推导轨迹）而非 "Chain of Thought" 或 "Reasoning Traces"——因为实际使用的中间 token 序列往往与真正的推理过程没有系统性的对应关系。

核心直觉：LLM 仅在"成品文本"上训练，缺少创作过程的草稿和中间步骤。但现实中——如高斯所言——"no self-respecting architect leaves the support structure in place once the edifice is complete!"

2.3.3 RL Post-Training 的过程形式化

以 DeepSeek R1-Zero/R1 为代表：在标准 LLM 预训练后，对具有自动可验证答案的问题进行 RL 后训练。

RL Post-Training 流程

  1. 生成轨迹：对每个问题 $Q$，LLM（作为 token-choosing policy $\pi_\theta$）生成 $k$ 条 completion：$\text{trace}_1, \text{trace}_2, \ldots, \text{trace}_k$，每条包含中间 token + 最终答案。
  2. 评分：仅依据最终答案正确性评分（不关心中间 token 质量）： $$R(\text{trace}_i) = \begin{cases} 1 & \text{if Sol}(\text{trace}_i, Q) \\ 0 & \text{otherwise} \end{cases}$$
  3. 策略更新：使用 policy gradient 选择性微调模型，提高高奖励轨迹中 token 的生成概率： $$\nabla_\theta J = \mathbb{E}_{\tau \sim \pi_\theta}\big[R(\tau) \cdot \nabla_\theta \log \pi_\theta(\tau)\big]$$
  4. 迭代：重复直至收敛或性能饱和。

💡 关键观察：作者指出，RL post-training 本质上是一个两阶段循环：生成轨迹 $\rightarrow$ 按成功加权 $\rightarrow$ 在自己的输出上选择性微调。这与单纯的 fine-tuning 并没有本质区别——"pure fine-tuning and RL approaches are not as different as might be initially assumed."

2.3.4 Distillation 及其意义

在 RL 训练模型（如 R1）的验证后输出上蒸馏（fine-tune）另一个 LLM，可以缩小甚至超越性能差距。这意味着：

  • RL 阶段可能并非必要——有了足够优质的 trace 数据，直接 SFT 就能达到类似效果
  • 公开数据集（如 Open Thoughts）已经出现，使得"跳过 RL 阶段"成为可能
  • 这进一步支持了论文的核心论点：本质上是"编译验证信号"，而非"学习推理"

3.3 SearchFormer / Stream of Search 的 Trace 验证实验

🔬 Trace 有效性验证实验设计

方法：在 SearchFormer 和 Stream of Search 的生成轨迹上，用原始生成算法（即用于产生训练数据的 A* 或搜索算法）重新检查每一条轨迹的操作合法性。

关键设置：测试问题与训练集独立同分布（identically distributed），排除了分布偏移导致的泛化失败。

发现的语义违规类型：

  • ❌ 从 Open List 中移除不存在的节点
  • ❌ 添加非邻居节点到 Open List
  • ❌ 关闭已在 Closed List 中的节点
  • ❌ 输出与搜索过程无关的最终答案
  • ⚠️ 即使 trace 中有语义违规，仍可能"蒙对"正确答案

4.2 五大核心结论

结论 1：LRM 是"更好的生成器"，而非"推理器"

o1 和 R1 相较于 vanilla LLM 的最大变化是：在同样的 generate-test 范式下，正确解的"密度"提高了（即 top-k 中含有更多正确的候选）。但它们仍然依赖外部验证器来最终确定正确性。最佳使用方式是将 o1 嵌入 LLM-Modulo 框架，由外部验证器提供正确性保证。

结论 2：Post-training LRM 在架构上与 LLM 无本质区别

R1 训练后得到的模型，其推理时的 token 生成方式与标准 LLM 完全一致——都是自回归地预测下一个 token。不同之处仅在于学到的概率分布 $P(x_t \mid x_{1:t-1})$ 发生了变化。所谓 "test-time scaling" 范式在 post-training-only LRM 上的应用，实际上与所有 LLM 都具备的"更长 completion = 更多计算"特性并无本质区别。

结论 3：推理被迭代编译为检索

这是本文最核心的理论贡献。通过 test-time verification → train-time RL → distillation 的三阶段编译流水线，验证器信号被一步步"编译"进基础 LLM，使其在越来越长的 "inference horizon" 上产生更好的 token 预测。这与以下证据一致：

  • CoT 的有效性高度依赖预训练数据中的"推理模式" (Stechly et al., 2024)
  • 在游戏规划中，internal planning（LLM 内部生成）vs. external planning（调用求解器）的性能差距 (Schultz et al., 2024)
  • Transformer 的 self-improvement 本质上是"将验证信号编译进模型" (Lee et al., 2025)

结论 4：中间 Token 不是"推理轨迹"

这是本文最大胆的论断。论文区分了两种概念：

  • Derivational Traces（推导轨迹）：模型在训练/推理中产生的原始中间 token 序列——可能包含 "hmm…"、"aha!"、"wait a minute" 等拟人化表达
  • Post-facto Explanations（事后解释）：o1 为用户提供的"经过消毒"的摘要版本——对人类而言，这类事后梳理很少能真实反映思考过程

SearchFormer/Stream of Search 的实验提供了直接证据：即使训练于算法生成的搜索轨迹，模型产生的 trace 中仍有大量语义违规——而这些违规的 trace 甚至可以"蒙对"最终答案。

⚠️ 潜在风险：故意让中间 token 看起来更"像人类思考"，可能利用用户的认知偏差来增强对错误答案的信任——"stylistically plausible ersatz reasoning traces seem ill-advised!"

结论 5：成本效益与商业模式困境

LRM 在训练和推理两端都引入了显著的额外成本。Test-time inference 尤其打破了 LLM 公司原本的商业模式——过去可以将模型成本前置（训练时），然后以低价出售 completion。而现在每次推理的成本都可能任意高。OpenAI 甚至对用户不可见的中间 token 收费，加剧了这一矛盾。在经济层面，LRM 是否优于更专门的经典求解器系统，仍是悬而未决的问题。

Section 5

锐评：作为 Reviewer 的视角

Overall: Weak Accept / Borderline — 重要的概念清理，但缺乏实验支撑

✅ 优势与贡献

1. 及时且必要的"去拟人化"干预

这是本文最大的价值。在 o1、R1 引发狂欢的当下，社区充斥着对 LRM "推理能力"的拟人化叙事。作者以严谨的术语区分（Derivational Traces ≠ Reasoning Traces）、尖锐的实验证据（SearchFormer 的语义违规）、以及清晰的哲学立场，提供了一剂急需的解毒剂。将 LRM 定性为 "Large Mumbling Models" 虽然刻薄，但确实切中要害。

2. "Reasoning Compiled into Retrieval" 的统一框架

这是极具解释力的概念框架。它将 test-time verification → RL post-training → distillation 串联为一条统一的"编译流水线"，优雅地解释了为什么 LRM 的表现提升呈现"在训练分布内越来越好，但泛化仍然脆弱"的模式。这与 Lee et al. (2025) 关于 transformer self-improvement 的理论分析高度吻合，增加了框架的可信度。

3. Skolem 函数的形式化

将 Prompt Augmentation 形式化为 Skolem 函数的学习问题，是一个巧妙的抽象。它揭示了 Zero-shot CoT、Few-shot CoT、Post-training 等技术之间的本质统一性——都是在学习"如何为任意任务 $T$ 找到合适的 $\text{PA}$"。同时，它暗示了一个有趣的未来方向：能否让第二 LLM 自由选择任意 token 词汇（而非局限于拟人化语言）来构建最优的 $\text{PA}$？这直接连接到了 adversarial attack 文献中"通用 token 串引导模型行为"的发现。

4. 对"隐藏中间 Token"商业实践的有力批判

作者指出 OpenAI 对用户隐藏 o1 的中间 token 但仍然收费的做法，不仅是商业不透明的问题，更是科学可验证性的问题——如果没有人能检查中间 token，我们就无法判断这些 token 是否具有真正的推理语义，抑或只是"风格上合理但语义上空洞"的填充物。DeepSeek R1 的开放性反而成为了对照：其 trace 可达"30 页文本"，但这种冗长本身也引发了对质量和必要性的质疑。

⚠️ 不足与局限

1. 缺少第一手实验数据

作为一篇 Perspective，这可以理解。但许多核心论断（如 "LRM 在 Mystery Blocksworld 上只是从零到有"、"更大规模实例仍会失败"）仅有定性描述，缺乏精确的数字支撑。读者无法自行判断"非平凡进展"到底意味着 5% 还是 45%。引用 PlanBench 原论文是合理的，但至少应提供一个摘要性的结果表。

2. 对"推理"的定义本身缺乏严格界定

论文花费大量篇幅论证 LRM 不是"真正推理"，但始终未给出一个可操作的"推理"定义。这在某种程度上削弱了论证的精确性——如果我们无法以可证伪的方式定义什么是推理，那么"LRM 不是在推理"就变成了一个不可证伪的立场，而非科学断言。论文可以引用哲学或认知科学中的 reasoning 操作性定义来弥补这一缺陷。

3. 对 Success Case 的解释力度不足

如果 LRM 只是"将验证信号编译为更好的检索"，那么为什么 o1 在 AIME 数学竞赛、CodeForces 编程等需要新颖组合的领域仍表现突出？论文将之归因为"预训练数据中可能已存在相似推理模式"，但这需要更直接的证据。一个更强的论证需要解释：验证信号的编译如何泛化到训练分布之外的组合新颖性。

4. SearchFormer / SoS 实验的代表性存疑

论文用 SearchFormer 和 Stream of Search 的 trace 违规来论证"LRM 的中间 token 可能不是真正推理"。但这两个模型使用的是域特定求解器轨迹训练的 encoder-decoder 架构——与 o1/R1 这类通用 LRM 存在显著差异。这一论证存在类比的外推风险。如果能对 R1 的公开 trace 进行类似的违规检查，论证将有力得多。

5. 对 Positive 方向的建议不够具体

论文在最后提出了一个有趣的方向——让第二 LLM 学习"任意词汇的 prompt augmentation 语言"（类似 AlphaZero 学习中间表示），但仅停留在猜测层面。考虑到作者群体的技术实力，一篇 Perspective 完全可以用半页篇幅勾勒一个具体的研究提案，而非止于 "this was our original speculation as to how o1 might be working"。

🔮 改进方向建议

  1. 对 R1 公开 Trace 进行系统审计：DeepSeek R1 的中间 token 是公开可见的。设计一个类似 SearchFormer 实验的审计框架——检查 R1 生成的 trace 中是否存在逻辑矛盾、语义违规、或与最终答案不一致的推理步骤——将极具说服力。
  2. 构建可控实验：在同一基座模型上，控制 trace 的内容（语义正确 vs. 语义错误但格式相同 vs. 无 trace），测量对下游任务性能的影响。这将直接回答"trace 的内容是否重要"这一核心问题。Li et al. (2025) 的发现——"结构而非内容决定了 CoT 的学习效果"——暗示答案可能令人意外。
  3. 定义"推理"的操作性标准：提出一个可验证的推理测试框架，例如：给定训练中未见过的组合规则，模型能否在零样本下正确应用？这比 PlanBench 的 Mystery 变体更具一般性。
  4. 探索非拟人化 PA 语言：作为一个具体的研究方向，训练一个独立的 "prompt optimizer" 模型，其输出 token 不必对人类可读——只需能提升主 LLM 的答案正确率。这将直接检验"推理是否需要人类可理解的中间步骤"这一假设。
  5. 成本效益的形式化分析：将 LRM 的 test-time compute 开销与经典求解器（如 Fast Downward）的计算成本进行定量比较，纳入准确性、延迟、和能源消耗维度。

6.1 "从 LLM 到 LMM"——本文最犀利的讽刺

"An uncharitable observer might even say that we went from LLMs to LMMs – Large Mumbling Models."

— Kambhampati, Stechly & Valmeekam (2025)

这句近乎刻薄的评论浓缩了本文的核心关切：当模型被训练来产生大量拟人化的喃喃自语——"hmm…", "aha!", "wait a minute…"——作为"推理"时，我们是否只是在训练一个更擅长说服人类自己很聪明的系统，而非一个真正更擅长解决问题的系统？

6.2 与 Adversarial Attack 文献的隐蔽联系

本文提出的"学习任意 token 词汇的 Prompt Augmentation 语言"这一方向，与 adversarial prompt engineering 文献有深刻的联系。Zou et al. (2023) 发现，可以通过优化找到对人类无意义但对模型有强烈引导作用的 token 序列。如果这样的优化是可能的方向——搜索"使模型输出更正确的 token 序列"而非"使模型输出有害内容"——那么"推理 trace"可能根本不需要是可读的语言。

这在某种程度上颠覆了当前 LLM 研究的基本隐喻：我们假设"推理"必须看起来像人类的推理，但也许对于硅基智能体，最优的"思考语言"根本不是自然语言。

6.3 R1-Zero 的"Aha Moment"——是涌现还是剧本？

DeepSeek 团队报告 R1-Zero 在 RL 训练中自发出现了 "Aha moment"——模型学会在 trace 中"自我修正"（如 "wait, let me reconsider"）。这一现象被广泛引用为"推理能力的涌现"。

然而从本文的框架来看，这也完全可以被解释为：RL 奖励的是"以正确答案结尾的任意 completion"。在预训练数据中，人类作者在纠错时经常使用 "wait"、"let me reconsider" 等短语——模型只是在 RL 过程中发现，生成这类模式增加了到达正确答案的概率。这与"涌现推理"的叙事有本质区别：它是在检索和执行一个学到的文本模式，而非进行真正的元认知。

6.5 对 AI Safety 的隐含启示

🚨 关键关注：如果 LRM 的中间 token 不是真正的推理轨迹，而只是风格上合理但语义上空洞的"合理性表演"，那么依赖这些 trace 进行 AI 安全监控（如 "让我们检查模型的思考过程以确认其安全性"）的做法是危险的。一个被训练来生成"看起来安全可信"的 trace 的模型，可能与一个实际进行安全推理的模型，在 trace 层面无法区分。这与 deceptive alignment 的担忧高度相关。
